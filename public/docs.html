Meeting Service API

A simple, RESTful API for creating and managing video meetings. Built for agents, services, and humans.

Base URL: http://localhost:3000
Default API Key: mk_default_test_key (for testing)

Table of Contents

1. Authentication 2. Meetings 3. Participant Controls 4. Meeting Settings 5. Real-time Events (Socket.IO) 6. WebRTC Signaling 7. Quick Start Guide 8. Error Handling

1. Authentication

All REST API requests require an API key passed via the x-api-key header.

Admin operations additionally require the x-admin-token header, which is returned when a meeting is created.

# Example headers
x-api-key: mk_default_test_key
x-admin-token: <admin-token-from-create>

2. Meetings

POST /api/meetings

Create a new meeting. Returns the meeting ID, admin token, and join URL.

Request Body

FieldTypeDescription
title optionalstringMeeting title (default: "Untitled Meeting")
muteOnJoin optionalbooleanAuto-mute new participants (default: false)
videoOffOnJoin optionalbooleanAuto-disable video for new participants (default: false)
maxParticipants optionalnumberMax participants allowed (default: 50)

Response

{
  "meetingId": "abc-defg-hij",
  "adminToken": "uuid-admin-token",
  "joinUrl": "/join/abc-defg-hij",
  "title": "Team Standup",
  "settings": {
    "muteOnJoin": false,
    "videoOffOnJoin": false,
    "maxParticipants": 50,
    "locked": false
  }
}

Example

curl -X POST http://localhost:3000/api/meetings \
  -H "Content-Type: application/json" \
  -H "x-api-key: mk_default_test_key" \
  -d '{"title": "Team Standup"}'
GET /api/meetings

List all active meetings.

Response

{
  "meetings": [
    {
      "meetingId": "abc-defg-hij",
      "title": "Team Standup",
      "createdAt": 1710000000000,
      "participantCount": 3
    }
  ]
}
GET /api/meetings/:meetingId

Get detailed information about a specific meeting, including all participants.

Response

{
  "meetingId": "abc-defg-hij",
  "title": "Team Standup",
  "createdAt": 1710000000000,
  "participantCount": 2,
  "participants": [
    {
      "participantId": "a1b2c3d4",
      "name": "Alice",
      "isMuted": false,
      "isVideoOff": false,
      "isScreenSharing": false,
      "joinedAt": 1710000001000
    }
  ],
  "settings": {
    "muteOnJoin": false,
    "videoOffOnJoin": false,
    "maxParticipants": 50,
    "locked": false
  }
}
DELETE /api/meetings/:meetingId

End a meeting and disconnect all participants. Admin

Headers

HeaderDescription
x-admin-tokenThe admin token returned from meeting creation
curl -X DELETE http://localhost:3000/api/meetings/abc-defg-hij \
  -H "x-api-key: mk_default_test_key" \
  -H "x-admin-token: your-admin-token"

4. Participant Controls

All participant control endpoints require x-admin-token.

POST /api/meetings/:meetingId/participants/:participantId/mute

Mute a specific participant. The participant receives a real-time event and their audio is disabled.

curl -X POST http://localhost:3000/api/meetings/abc-defg-hij/participants/a1b2c3d4/mute \
  -H "x-api-key: mk_default_test_key" \
  -H "x-admin-token: your-admin-token"
POST /api/meetings/:meetingId/participants/:participantId/unmute

Unmute a specific participant.

POST /api/meetings/:meetingId/participants/:participantId/kick

Remove a participant from the meeting. They will be disconnected immediately.

Request Body

FieldTypeDescription
reason optionalstringReason for removal shown to the participant
POST /api/meetings/:meetingId/mute-all

Mute all participants in the meeting at once.

POST /api/meetings/:meetingId/invite

Generate an invite link for a new participant.

Request Body

FieldTypeDescription
name optionalstringPre-filled name for the invitee

Response

{
  "joinUrl": "/join/abc-defg-hij?invite=x1y2z3w4&name=Bob",
  "inviteToken": "x1y2z3w4"
}

5. Real-time Events (Socket.IO)

Connect to the Socket.IO server at the same base URL. Events enable real-time participant management and media signaling.

Client → Server Events

EventPayloadDescription
join-meeting{ meetingId, name, isAdmin, adminToken }Join a meeting room
signal:offer{ to, offer }Send WebRTC offer to a participant
signal:answer{ to, answer }Send WebRTC answer
signal:ice-candidate{ to, candidate }Send ICE candidate
media:toggle-audio{ isMuted }Notify mute state change
media:toggle-video{ isVideoOff }Notify video state change
media:screen-share{ isScreenSharing }Notify screen share state
chat:message{ text }Send a chat message

Server → Client Events

EventPayloadDescription
joined{ participantId, participants, settings, title, isAdmin }Confirmation of successful join with existing participants
participant:joined{ participantId, name, isMuted, isVideoOff, isAdmin }A new participant joined
participant:left{ participantId, name }A participant left
participant:updated{ participantId, ...changes }Participant state changed (mute/video/screen)
signal:offer{ from, offer }Incoming WebRTC offer
signal:answer{ from, answer }Incoming WebRTC answer
signal:ice-candidate{ from, candidate }Incoming ICE candidate
admin:mute{}You were muted by admin
admin:unmute{}You were unmuted by admin
admin:kick{ reason }You were removed from the meeting
meeting:ended{ reason }Meeting was ended by admin
meeting:all-muted{}All participants were muted
meeting:settings-updated{ ...settings }Meeting settings changed
chat:message{ from, name, text, timestamp }Chat message received
error{ message }Error (meeting not found, locked, full)

6. WebRTC Flow

The service uses a mesh topology for WebRTC peer connections. Each participant connects directly to every other participant.

Connection Flow

1. Client A joins → receives list of existing participants
2. Client A creates RTCPeerConnection for each existing participant
3. Client A sends offer via signal:offer
4. Client B receives offer → creates answer → sends via signal:answer
5. Both exchange ICE candidates via signal:ice-candidate
6. Direct peer-to-peer media flows between A and B

Media Tracks:
- Audio: microphone (can be muted/unmuted)
- Video: camera (can be toggled on/off)
- Video: screen share (replaces camera track temporarily)
- Video: blur filter (processed canvas stream replaces camera track)

STUN Servers Used

stun:stun.l.google.com:19302
stun:stun1.l.google.com:19302

For production, add TURN servers for participants behind strict NATs/firewalls.

7. Quick Start Guide

For Agents / Services (API-only)

# 1. Create a meeting
RESPONSE=$(curl -s -X POST http://localhost:3000/api/meetings \
  -H "Content-Type: application/json" \
  -H "x-api-key: mk_default_test_key" \
  -d '{"title": "Agent Meeting"}')

MEETING_ID=$(echo $RESPONSE | jq -r '.meetingId')
ADMIN_TOKEN=$(echo $RESPONSE | jq -r '.adminToken')

# 2. Share the join link with participants
echo "Join: http://localhost:3000/join/$MEETING_ID"

# 3. Monitor participants
curl -s http://localhost:3000/api/meetings/$MEETING_ID \
  -H "x-api-key: mk_default_test_key" | jq

# 4. Mute a noisy participant
curl -X POST http://localhost:3000/api/meetings/$MEETING_ID/participants/PARTICIPANT_ID/mute \
  -H "x-api-key: mk_default_test_key" \
  -H "x-admin-token: $ADMIN_TOKEN"

# 5. End the meeting
curl -X DELETE http://localhost:3000/api/meetings/$MEETING_ID \
  -H "x-api-key: mk_default_test_key" \
  -H "x-admin-token: $ADMIN_TOKEN"

For Humans (Web UI)

1. Open http://localhost:3000 in your browser
2. Enter a meeting title and your name, click "Create Meeting"
3. Share the meeting link with others
4. Use the toolbar to: toggle mic, camera, share screen, blur background
5. Open the People panel to manage participants (admin only)

For Programmatic Clients (Socket.IO)

const { io } = require("socket.io-client");

const socket = io("http://localhost:3000");

socket.emit("join-meeting", {
  meetingId: "abc-defg-hij",
  name: "Bot",
  isAdmin: false,
  adminToken: null,
});

socket.on("joined", (data) => {
  console.log("Joined as", data.participantId);
  console.log("Existing participants:", data.participants);
});

socket.on("participant:joined", (p) => {
  console.log(p.name, "joined");
});

socket.on("chat:message", ({ name, text }) => {
  console.log(`${name}: ${text}`);
});

// Send a chat message
socket.emit("chat:message", { text: "Hello from the bot!" });

8. Error Handling

StatusErrorDescription
401Invalid or missing API keyMissing or wrong x-api-key header
403Admin token requiredMissing or wrong x-admin-token for admin endpoints
404Meeting not foundMeeting ID doesn't exist or has ended
404Participant not foundParticipant ID doesn't exist in the meeting

Socket.IO errors are emitted via the error event with a { message } payload.

Production Notes:
• Add TURN servers (e.g. Twilio, Xirsys) for reliable NAT traversal
• Replace in-memory storage with a database (Redis/PostgreSQL)
• Generate proper API keys with rate limiting
• Add TLS/HTTPS (required for WebRTC in production)
• Consider an SFU (Selective Forwarding Unit) for meetings with 6+ participants
